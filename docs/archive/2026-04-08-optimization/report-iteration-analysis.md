# 新闻分析系统分层检查与迭代优化方案

**分析日期**: 2026-04-08  
**目标报告**: 【2026-04-07】全球权威新闻分析报告.zip  
**分析范围**: 全工作流（数据采集 → 解析清洗 → 分析生成 → 输出交付）

---

## 一、理解确认

### 1.1 项目核心目的

基于第一性原理推导，该系统的**原始目的**是：

> **为投资决策、政策把握、科技跟踪三大场景提供每日全球权威新闻的自动化采集、智能分析和结构化报告生成服务。**

核心价值链：
```
多源权威新闻 → AI智能解析评分 → 分层报告（简报+深度） → 邮件推送决策者
```

### 1.2 报告使用场景

| 场景 | 使用者 | 需求 | 当前交付物 |
|------|--------|------|-----------|
| 移动快速浏览 | 高管/决策者 | 当日TOP10，快速掌握要点 | 简要报告（Markdown） |
| 深度研究分析 | 分析师/研究员 | 领域细分、历史关联、趋势判断 | 深度报告（PDF，按领域） |
| 历史回溯对比 | 策略部门 | 长期趋势、事件演化 | 数据库存储 + 向量索引 |

### 1.3 技术栈与架构

```
数据采集: RSS采集 + 定向爬虫
数据处理: 12步流水线（字段标准化 → AI合并处理 → 向量化 → 热度评分 → 存储）
数据存储: SQLite（单一数据源）
AI模型: DeepSeek(深度分析) + 豆包(快速筛选) + 自动熔断切换
报告生成: 双轨制（简报TOP10 + 深度领域分析）
部署运行: GitHub Actions（每日3次采集 + 1次报告）
```

### 1.4 现有质量评估

基于 `PROJECT_EVALUATION_2026-04-07.md`：
- 功能完整性: ★★★★☆ (核心功能完备)
- 代码质量: ★★★☆☆ (4处P0运行时崩溃风险，16个死代码模块已清理)
- 配置架构: ★★★★☆ (评分参数已配置化)
- 文档覆盖: ★★★★☆ (5层文档体系)

## 二、分层检查表

### 2.1 数据输入层（RSS采集）

| 检查项 | 现状 | 问题 | 风险等级 |
|--------|------|------|---------|
| **数据来源可靠性** | 45KB的sources.yaml，Tier 1/2/3分层 | Tier 3垂直媒体可能存在质量参差不齐 | 🟡 中 |
| **时效性** | 增量采集 + 智能回溯 | 补救采集时区比较存在缺陷（已修复） | 🟢 低 |
| **完整性** | 多源采集 + 遗漏检测 + 补救采集 | 任务取消导致raw_news残留611条（已清理） | 🟢 低 |
| **偏误风险** | RSS源以西方主流媒体为主 | 中文/非英语源覆盖不足，可能存在视角偏误 | 🔴 高 |

**关键矛盾**:
- 系统设计目标是"全球权威"，但RSS源配置中西方媒体占比过高，可能产生系统性视角偏误
- 补救采集机制依赖`rss_rollover_hours`，但不同源的feed滚动窗口差异大

---

### 2.2 解析清洗层（12步流水线）

| 检查项 | 现状 | 问题 | 风险等级 |
|--------|------|------|---------|
| **新闻分类** | 轻量级8类分类器（政治/经济/科技/军事/社会/文化/体育/娱乐） | 置信度阈值配置已生效，但分类错误会传递 | 🟡 中 |
| **权重计算** | 4维评分（source/influence/value/heat），权重可配置 | heat_score依赖热榜API，可能因API失败而缺失 | 🟡 中 |
| **去重逻辑** | 5层去重（raw_news唯一约束 → 内存去重 → processed_news查询 → 存储前检查 → 单条检查） | 任务取消导致processed_news未更新（已修复save-always） | 🟢 低 |
| **摘要生成** | AI单次调用合并处理（翻译+摘要+5W1H+评分） | 熔断器触发时会跳过批次，导致数据丢失 | 🔴 高 |

**关键矛盾**:
- AI处理是瓶颈：单次API调用包含翻译+摘要+5W1H+评分，任一部分失败会导致整条新闻丢弃
- 熔断器设计初衷是避免浪费CI时间，但连续3次致命错误后跳过剩余批次，可能导致当日新闻覆盖不完整

---

### 2.3 分析生成层（报告生成）

| 检查项 | 现状 | 问题 | 风险等级 |
|--------|------|------|---------|
| **简报生成** | TOP10分中国/国外，按final_score排序 | 仅展示得分，无趋势对比（与昨日/上周） | 🟡 中 |
| **深度报告** | 仅处理政治/经济/科技3个领域 | 军事/社会/文化/体育/娱乐被忽略（数据量判断？） | 🔴 高 |
| **事件聚类** | AI聚类 → TOP5事件 | 聚类质量依赖AI，无人工校验 | 🟡 中 |
| **历史关联** | BGE-M3向量索引 + TF-IDF fallback | 历史关联分析深度依赖原文获取成功率 | 🟡 中 |

**关键矛盾**:
- 深度报告硬编码只处理3个领域（政治/经济/科技），其他领域新闻被排除，与"全球"目标不符
- 报告缺乏时间维度对比（当日 vs 近期均值），难以判断"异常/趋势"
- 无置信度/可靠性指标，读者无法判断AI分析的置信水平

---

### 2.4 输出交付层（报告交付）

| 检查项 | 现状 | 问题 | 风险等级 |
|--------|------|------|---------|
| **格式规范** | Markdown(简报) + PDF(深度) | PDF中文字体依赖系统安装，可能渲染异常 | 🟡 中 |
| **可读性** | 结构化表格 + AI生成文本 | 大段AI文本缺乏层次感，关键结论不突出 | 🟡 中 |
| **可行动建议** | 深度报告含投资分析（可选） | 建议粒度粗，缺乏具体标的时间窗口 | 🔴 高 |
| **反馈闭环** | 无 | 无用户反馈收集机制，无法持续优化 | 🔴 高 |

**关键矛盾**:
- 报告是"只读"输出，无用户互动机制（评分、纠错、需求反馈）
- 邮件推送后无阅读/点击追踪，无法评估报告价值
- 无A/B测试机制，无法验证不同报告格式的效果

## 三、异常解决方案（按优先级排序）

### P0: 关键功能缺陷（立即修复）

#### P0-1: 深度报告领域覆盖不全

**问题**: 硬编码只处理政治/经济/科技，军事/社会/文化/体育/娱乐被排除

**方案**:
```python
# 修改 report_generator.py:generate_depth_reports()
# 原代码
domains = ['政治', '经济', '科技']

# 改为配置驱动
domains = self.config.get('depth_report_domains', 
                         ['政治', '经济', '科技', '军事', '社会'])
# 或动态判断：只要有新闻的领域都生成报告
```

**回退策略**: 保留现有3领域作为默认配置，允许通过环境变量扩展

#### P0-2: 熔断器跳过批次导致数据丢失

**问题**: 连续3次AI致命错误后，剩余批次全部跳过

**方案**:
```python
# 改进熔断器逻辑：跳过改为降级
if self.is_circuit_open():
    # 不降入度，而是使用轻量级本地处理
    logger.warning("熔断器触发，切换到轻量级处理模式")
    # 保留原文 + 基础评分（无翻译/摘要）
    result = self.lightweight_fallback_process(news)
```

### P1: 重要优化（短期实施）

#### P1-1: 增加时间维度对比

**问题**: 报告缺乏与历史数据的对比，难以识别趋势

**方案**:
```python
# 在简报中增加趋势列
report_lines.append("| 排名 | 事件标题 | 当日得分 | 近7天均值 | 趋势 |")
# 趋势：↑显著高于均值 / →持平 / ↓低于均值
```

#### P1-2: 报告置信度标识

**问题**: AI分析结果无置信度指示

**方案**:
```python
# 使用 accuracy_score 分级
if accuracy_score >= 0.8:
    confidence_label = "【高置信度】"
elif accuracy_score >= 0.6:
    confidence_label = "【中置信度】"
else:
    confidence_label = "【低置信度-AI兜底】"
```

#### P1-3: 信源视角偏误平衡

**问题**: 西方媒体占比过高，视角单一

**方案**:
1. 增加 Tier 1 中文/非英语源（如新华社、Al Jazeera、RT等）
2. 在报告中增加"信源分布"统计，让读者了解视角构成
3. 同一事件多源对比展示

## 四、架构优化建议

### 4.1 当前架构局限

```
当前架构：线性流水线
RSS → 采集 → AI处理 → 存储 → 报告生成 → 邮件

局限：
1. 单点故障：AI处理失败 = 数据丢失
2. 无反馈闭环：无法持续改进
3. 静态配置：领域/权重调整需改代码
```

### 4.2 演进架构建议

```
目标架构：弹性流水线 + 反馈闭环

         ┌──────────────────────────────────────┐
         │           质量门禁层                  │
         │  (人工抽检 + 自动校验 + 用户反馈)      │
         └──────────────┬───────────────────────┘
                        │
    ┌───────────────────┼───────────────────┐
    ▼                   ▼                   ▼
┌─────────┐      ┌────────────┐      ┌──────────┐
│ 快速通道 │      │  标准处理   │      │ 深度处理  │
│(本地规则)│      │  (AI FILTER)│      │ (AI ANALYSIS)
└────┬────┘      └──────┬─────┘      └────┬─────┘
     │                  │                 │
     └──────────────────┼─────────────────┘
                        ▼
              ┌──────────────────┐
              │   自适应路由      │
              │  (根据质量/时效性) │
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │   统一存储层      │
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │   多格式报告生成  │
              │ (个性化 + A/B测试)│
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │   多渠道交付      │
              │(邮件/钉钉/飞书/Web)│
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │   反馈收集分析    │
              └──────────────────┘
```

### 4.3 核心改进点

| 改进 | 当前 | 目标 | 实施复杂度 |
|------|------|------|-----------|
| **多层级处理** | 单层AI处理 | 快速通道 + 标准 + 深度 | 中 |
| **质量门禁** | 仅DataValidator | 增加人工抽检 + 用户反馈 | 低 |
| **自适应路由** | 固定流程 | 根据新闻重要性动态选择处理深度 | 高 |
| **多渠道交付** | 仅邮件 | 邮件 + IM + Web | 中 |
| **反馈闭环** | 无 | 评分 + 纠错 + 需求收集 | 低 |

## 六、结论与建议

### 6.1 核心发现

1. **数据输入层**: 西方媒体视角偏误是最大风险，建议立即增加多元信源
2. **处理分析层**: 熔断器设计有缺陷，建议改为降级而非跳过
3. **生成输出层**: 领域覆盖不全，建议配置化扩展
4. **交付反馈层**: 缺乏反馈闭环是最大短板，建议优先实施Web反馈

### 6.2 第一性原理反思

原始目的：服务"投资分析、制度掌握、科技发展"三大决策方向

关键问题：
- 当前报告是否能有效支持这三大决策？
- 缺少量化指标验证报告价值（阅读率、反馈率、决策引用率）

建议：
- 建立"报告价值评估"机制（最简单的：邮件点击追踪）
- 定期（季度）向核心用户收集反馈，验证方向正确性

### 6.3 下一步行动

1. **确认方案优先级**：请确认P0/P1/P2的优先级是否符合预期
2. **提供用户反馈**：如有核心用户反馈，请提供以指导优化方向
3. **确认信源扩展**：是否同意增加中文/非英语Tier 1源？推荐清单？
4. **确认架构演进**：是否同意向"弹性流水线 + 反馈闭环"演进？
